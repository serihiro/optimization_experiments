# AGENTS.md

## 最適化テーマを横展開するための標準手順

このリポジトリで「naive実装 → 段階的最適化 → benchmark比較 → report作成」を繰り返すときは、以下を基本フローにする。

1. **テーマ定義**
   - 対象問題（例: Mandelbrot, 畳み込み, 行列演算）を決める。
   - 入出力仕様と正解条件（許容誤差含む）を先に固定する。

2. **実装バリエーション設計**
   - `naive`（可読性優先）を最初に作る。
   - そこから 2〜4 段階の最適化案を作る（例: ループ整理、SIMD、アンロール、キャッシュ最適化）。
   - 各段階は「何を変えたか」が1つに近い粒度になるように分割する。

3. **正しさ検証の仕組みを作る**
   - 全実装で同一入力を使う。
   - checksum / 誤差指標 / 参照実装比較で一致確認する。
   - benchmark前に正しさが崩れていないことを必ず確認する。

4. **benchmark環境を固定する**
   - 問題サイズ、反復回数、コンパイラオプション、実行回数を明記する。
   - `best` と `avg` の両方を記録する。
   - 可能ならウォームアップを入れる。

5. **結果の整理と考察**
   - 実装ごとの性能表（実行時間、speedup、正しさ）を作る。
   - 「なぜ速くなった/ならなかったか」をアーキテクチャ観点（分岐、メモリ、SIMD効率）で書く。
   - 次の改善案を 2〜4 個書く。

6. **成果物を残す**
   - テーマ別ディレクトリに `Makefile` と `report.md` を置く。
   - `README.md` の Contents を更新する。

## 実装時の推奨ディレクトリ構成

```text
<theme>/
  benchmark.c            # 複数実装 + 計測
  Makefile               # build/run/clean
  report.md              # 比較表と考察
```

## 再現コマンドのテンプレート

```sh
cd <theme>
make run
```

必要に応じて `RUNS`, 問題サイズ, コンパイラフラグをコードまたはMakefileで変更する。

## Markdown 数式記法ルール

- このリポジトリの Markdown で数式を書くときは、**GitHub 表示互換**の記法を使う。
- インライン数式は `$ ... $` で囲む。
- 複数行・別行で目立たせる数式は次の fenced math block を使う。

```markdown
```math
E = mc^2
```
```

- `\[ ... \]` や `\( ... \)` は使わず、上記の `$` / ````math` ブロックに統一する。
