# GEMM最適化レポート（cache locality + blocking）

## 概要
この実験では、同じ行列積 `C = A x B` を次の2種類で実装し、性能比較を行いました。

- `naive.c`: もっとも素直な3重ループ（`i-j-k`）
- `optimized.c`: 次の最適化を適用
  - `B` を事前に転置（`B^T` を作成）して連続アクセス化
  - ブロッキング（`BLOCK_SIZE=64`）
  - 内側ループを4要素ずつ手動アンロール

## ベンチマーク方法
`benchmark.py` で以下を自動実行します。

1. `make clean && make` で2つのバイナリをビルド
2. 行列サイズ `N = [256, 384, 512]` でそれぞれ3回実行
3. 平均GFLOPSを計算
4. `checksum` を比較して結果の一致を確認

実行コマンド:

```bash
./gemm_cache_optimization/benchmark.py
```

## 結果

| N | naive GFLOPS | optimized GFLOPS | speedup |
|---:|---:|---:|---:|
| 256 | 1.259 | 1.661 | 1.32x |
| 384 | 1.141 | 1.793 | 1.57x |
| 512 | 0.534 | 1.972 | 3.69x |

詳細な実行結果（平均時間・checksum含む）は `results.json` に保存されます。

## 考察
- `naive` の `i-j-k` ループでは、`B[k][j]` が列方向アクセスになるためキャッシュ効率が悪化しやすいです。
- `B` を転置して `B^T[j][k]` を読むと、内側ループで `A[i][k]` と `B^T[j][k]` がどちらも連続アクセスになり、メモリアクセス効率が改善します。
- さらにブロッキングでワーキングセットを小さく保つことで、サイズが大きいほど改善率が大きくなる傾向が見られました（`N=512` で 3.69x）。
