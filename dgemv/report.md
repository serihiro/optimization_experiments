# dgemv ベンチマークレポート

## 実行環境
- 日時: 2026-03-25 (UTC)
- コンパイラ: `gcc`
- ビルド設定: `-O0 -std=c11 -Wall -Wextra`（`naive`）
- 並列実装ビルド設定: `-fopenmp`（Makefileの `embarassingly_parallel.o` ターゲット）
- 問題サイズ: `N = 10240`
- 実行コマンド: `make -C dgemv clean && make -C dgemv execute_all`

## ベンチマーク結果
`execute_all` の1回実行結果を記録しています。

| 実装 | elapsed time [sec] |
|---|---:|
| naive | 0.598488 |
| embarrassingly_parallel | 1.325799 |

## 比較と考察
- 今回は `naive` が最速で、OpenMP版 (`embarrassingly_parallel`) は約 **2.22 倍遅い**結果でした。
- 並列版が遅い主因として、以下が考えられます。
  - 外側ループ (`i`) ごとに並列領域を生成しているため、スレッド起動・同期オーバーヘッドが大きい。
  - `N=10240` で1行あたりの計算量はあるものの、`-O0` で最適化が無効のため並列化の利得が出にくい。
  - `tmp_sum` が `int` のため、演算の型変換コストや実装上の制約が入りやすい。
- 互換性面では、古いコードでもこの環境では実行可能でした。
- ただし `Makefile` には `CFALGS` という変数名のtypoがあり、並列版のコンパイル時に意図した警告/標準指定フラグが効いていませんでした。今回の再現性向上のため、このtypoは `CFLAGS` に修正しました。

## 補足
- ベンチマークは1回実行のため、厳密比較には複数回実行（中央値/平均）での再測定を推奨します。
