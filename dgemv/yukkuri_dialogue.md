# dgemv 実験のゆっくり解説（霊夢×魔理沙）

## 1) この課題は何をするコード？

**霊夢：**
`dgemv` は `dgemm` と何が違うの？

**魔理沙：**
`dgemv` は **行列×ベクトル** だ。
```math
y = A x
```

ここで `A` は `N \times N`、`x` は長さ `N`、`y` も長さ `N`。`dgemm` より計算量は少ないが、メモリ帯域の影響を受けやすい。

---

## 2) 単純実装（naive）の数式

**霊夢：**
数式は？

**魔理沙：**
基本はこれ。
```math
y_i = \sum_{j=0}^{N-1} A_{ij} x_j
```
各行 `i` ごとに内積を取るだけだ。

**霊夢：**
式は簡単だけど、実装では `i` と `j` の回し方や並列化の粒度が大事そう。

**魔理沙：**
その読みは正しいぜ。

---

## 3) それぞれの最適化アプローチ

**霊夢：**
このディレクトリではどんな最適化を比較してるの？

**魔理沙：**
主に2種類だ。

### `naive.c`
- 素直な逐次実装。
- オーバーヘッドが少なく、基準として使いやすい。

### `embarrassingly_parallel.c`
- OpenMP でループを並列化。
- 狙いは複数コアを使って処理時間短縮。
- ただし、並列領域の作り方次第で「スレッド管理コスト」が勝ってしまい、逆に遅くなる場合もある。

**霊夢：**
「並列化すれば必ず速い」じゃないってことね。

**魔理沙：**
そう。`dgemv` は1要素あたりの計算密度がそこまで高くないから、
- 並列化の粒度
- スレッド生成/同期の位置
- コンパイラ最適化レベル
を丁寧に設計しないと逆効果が起こるんだ。
