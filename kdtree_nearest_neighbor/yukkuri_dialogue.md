# KD-tree nearest neighbor をゆっくり解説

## 登場人物
- **ゆっくり霊夢（先生役）**
- **ゆっくり魔理沙（先生役）**

---

**霊夢:** 今日は「最近傍探索」を最適化するわよ。テーマは `brute force`、`kd-tree`、`SIMD distance` の 3 本立て。

**魔理沙:** LLM embedding search の基礎構造にも近い話だな。クエリベクトルに最も近いベクトルを探す処理だ。

**霊夢:** まず基準実装は単純明快。全データ点との距離二乗を計算して、最小値を選ぶ。

**魔理沙:** これは計算量が $O(N \cdot D)$。点数 $N$ が増えると、クエリごとの時間が直線的に増える。

**霊夢:** そこで `SIMD distance`。1 点との距離計算を AVX2 で 8 要素ずつ処理して、1 比較あたりのコストを下げる。

**魔理沙:** ただし全探索自体は変わらないので、オーダーは同じ。改善は「定数倍」だな。

**霊夢:** 次に `kd-tree`。空間を軸ごとに分割して木構造にし、探索で不要な枝を切る。

**魔理沙:** 枝刈り判定は「分割超平面までの距離」で行う。今の最良距離より遠い側の枝はスキップ可能だ。

**霊夢:** つまり `kd-tree` は「比較回数そのもの」を減らす。これは SIMD と違う効き方ね。

**魔理沙:** LLM embedding search の実務では次元が 384～1536 と高く、KD-tree が苦手になることもある。

**霊夢:** だからこそ大規模検索では HNSW や IVF-PQ のような近似最近傍法がよく使われるのよ。

**魔理沙:** でも学習順序としては、
1. 全探索（正解基準）
2. SIMD（演算高速化）
3. インデックス（候補削減）
の流れが理解しやすい。

**霊夢:** 今回の実験でも、3 実装すべてで同じ入力を使い、`brute_force` と結果照合して正しさを担保しているわ。

**魔理沙:** 最適化では速度だけ追うと壊しやすい。正しさ検証を先に固定するのが重要だな。

**霊夢:** まとめると、
- SIMD: 1 比較を速くする
- KD-tree: 比較回数を減らす
- embedding search 本番: さらに ANN でスケールさせる
という対応関係になる。

**魔理沙:** 次は KD-tree の leaf だけ SIMD 化して「枝刈り + SIMD」を合体させると、さらに面白くなるぜ。
